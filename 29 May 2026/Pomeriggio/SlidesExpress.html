Node.js: Runtime Environment e Architettura I/O

  • Node.js si definisce formalmente come un runtime environment asincrono basato sul motore JavaScript V8.
  • Consente l'esecuzione nativa di JavaScript sul lato server, garantendo l'accesso diretto a risorse di sistema altrimenti precluse.
  • Fornisce interfacce dirette per la manipolazione del File System e la gestione di socket di rete.
  • Implementa un'architettura di tipo Event-Driven coadiuvata da logiche di Non-Blocking I/O.
  • È strutturato per ottimizzare e sostenere migliaia di connessioni simultanee su un unico thread.
  • Il modulo nativo core http richiede tuttavia una prolissa stesura di codice ripetitivo per il parsing manuale di URL, intestazioni e flussi raw di byte.

Inizializzazione Progetto e Installazione Dipendenze (NPM)

  • Prima di scrivere codice, è necessario inizializzare l'ambiente tramite Node Package Manager (NPM).
  • Il comando npm init -y genera il file package.json, indispensabile per tracciare il progetto e le sue dipendenze.
  • Le librerie fondamentali non sono preinstallate. È necessario eseguire l'installazione di Express, Mongoose e i middleware utili: 
npm install express mongoose dotenv cors

Express.js: Il Framework Minimalista per Node.js

  • Express.js costituisce lo standard industriale strutturale adibito allo sviluppo di applicazioni web e API RESTful all'interno dell'ecosistema Node.js.
  • Garantisce caratteristiche di flessibilità estrema essendo un framework di tipo minimalista e unopinionated.
  • Non vincola lo sviluppatore a una rigida predefinizione dell'alberatura dei file di progetto.
  • Funzionalità Core fornite dal framework:
    • Strumenti di astrazione semplificata per la gestione delle rotte (Routing).
    • Meccanismi ad architettura Middleware per il monitoraggio e controllo dei flussi operativi.
    • Sistemi interni di parsing dei payload (corpi delle richieste HTTP in entrata).
    • Interfacce immediate per la manipolazione e l'invio delle risposte HTTP.

Ambiente di Sviluppo: Nodemon e Variabili d'Ambiente

  • Nodemon: Per evitare di dover riavviare manualmente il server ad ogni modifica al codice, si utilizza nodemon (npm install -g nodemon). Questo tool riavvia in automatico il processo demone quando rileva salvataggi.
  • Variabili d'Ambiente (.env): L'ambiente (environment) è il contesto in cui viene eseguito il server. Informazioni sensibili come credenziali del database o porte di rete non vanno mai scritte hardcoded nel codice sorgente. Si usa il pacchetto dotenv per leggerle da un file nascosto chiamato .env. 
require('dotenv').config();
const PORT = process.env.PORT || 3000;

Architettura Client-Server e API REST

  • La comunicazione fluttua mediante lo scambio deterministico di pacchetti HTTP (Request/Response) tra un client generico e l'istanza del server.
  • Una HTTP Request si compone di: un Metodo (Verbo), un Endpoint (URL), una serie di Headers e un Body opzionale.
  • I Verbi HTTP regolano i criteri per la persistenza e manipolazione dei dati nelle operazioni CRUD:
    • GET: Interrogazione e recupero di record (sola lettura, nessun body).
    • POST: Trasmissione payload per l'inserimento e la creazione di un nuovo record.
    • PUT / PATCH: Mutazione o sovrascrittura parziale/totale di record esistenti.
    • DELETE: Rimozione permanente di una risorsa specifica.
  • Il server intercetta la richiesta, esegue l'elaborazione e restituisce una HTTP Response provvista di status code standardizzato (es. 200 OK, 404 Not Found) e payload JSON.

Sicurezza Base: CORS (Cross-Origin Resource Sharing)

  • Per motivi di sicurezza, i browser bloccano di default le richieste HTTP verso un server che risiede su un dominio o una porta diversa (es. Frontend in React su porta 5173, Backend Express su porta 3000).
  • Questo blocco va disabilitato esplicitamente sul backend installando e dichiarando il middleware cors, permettendo così al client web di comunicare regolarmente con le nostre API. 
const cors = require('cors');
app.use(cors()); // Abilita le richieste da altri domini

Configurazione Minimale di un Server Express

  • Dichiarazione sintattica fondamentale per l'istanziazione di un'applicazione Express e l'attivazione del processo demone in ascolto su una porta di rete: 
const express = require('express');
const cors = require('cors');
require('dotenv').config();

const app = express();
const PORT = process.env.PORT || 3000;

app.use(cors());

// Il metodo listen avvia il server e blocca la porta specificata
app.listen(PORT, () => {
    console.log("Processo server attivo sulla porta di rete:", PORT);
});

Gestione delle Rotte (Routing) e Oggetti Req/Res

  • Il routing associa un endpoint e un metodo HTTP a una specifica funzione di callback incaricata di elaborare i parametri di Request (req) e Response (res): 
// Definizione di endpoint GET con serializzazione JSON automatica
app.get('/api/stato', (req, res) => {
    res.json({ server: "online", timestamp: Date.now() });
});

// Gestione di endpoint dinamici mediante l'uso di Route Parameters (:id)
app.get('/api/utenti/:id', (req, res) => {
    const idUtente = req.params.id; // Estrazione del parametro dinamico dall'URL
    res.json({ id: idUtente, recuperato: true });
});

Ricezione di Payload nel Body (Richieste POST)

  • Per specifica, Express non effettua la decodifica automatica dei corpi delle richieste. È imperativo registrare preventivamente il middleware di parsing del formato JSON: 
// Configurazione del middleware nativo per l'intercettazione e il parsing di JSON
app.use(express.json());

app.post('/api/prodotti', (req, res) => {
    const nuovoProdotto = req.body; // Accesso all'oggetto JSON già decodificato
    console.log("Oggetto inserito:", nuovoProdotto);
    
    // Restituzione dello status HTTP 201 (Created) con annesso payload di conferma
    res.status(201).json(nuovoProdotto);
});

Completamento del CRUD (PUT e DELETE)

  • Per avere una gestione dati completa, servono anche gli endpoint per aggiornare e rimuovere record dal sistema.
  • L'operazione di PUT riceve l'ID del record come parametro e i nuovi dati all'interno del body della richiesta.
  • L'operazione di DELETE riceve solo l'ID da eliminare, rispondendo tipicamente con uno status code "204 No Content" e senza payload. 
// PUT: Aggiornamento record
app.put('/api/utenti/:id', (req, res) => {
    const id = req.params.id;
    const updateDati = req.body;
    res.json({ id: id, aggiornatoCon: updateDati });
});

// DELETE: Rimozione record
app.delete('/api/utenti/:id', (req, res) => {
    const id = req.params.id;
    res.status(204).send(); // 204 No Content (eliminato con successo)
});
